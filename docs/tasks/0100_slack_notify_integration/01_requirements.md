## 1. 背景と目的

### 1.1 背景

Slack 通知メッセージは、既存のユニットテストではテストダブルや slog キャプチャによって検証しており、実際の Slack Webhook へのリクエスト送信と Slack 上での表示品質は開発者が手動で確認するしかない状態にある。

特に以下の点が未検証である。

- 実 Slack サーバへの送信が正常に完了すること（Webhook URL の疎通確認）。
- Slack が受信したメッセージが、期待どおりの書式で描画されること。

### 1.2 目的

1. **主目的**: failure を含む TLS-RPT メールから failure を検出し、実際の Slack Webhook に即時アラートを送信できることを確認する統合テストを追加する。
2. **副次的目的**: 開発者が Slack 上でメッセージ書式を目視検証するための再現可能な手順を提供する。

---

## 2. スコープ

### 対象範囲（In Scope）

- 専用ビルドタグ（`slack_notify`）付き統合テストの新規作成
  - `testdata/tlsrpt_failure.eml` を入力とする
  - EML パース → failure 検出 → 実 Slack Webhook へのアラート送信という一連のフローを実行する
  - 対象 Webhook URL は環境変数で指定し、未設定時はテストをスキップする
  - 本番 `fetch` が生成する解析結果ファイル（`tlsrpt.json`）などの永続アーティファクトを残さない（繰り返し・並列実行を阻害しない）
- 本テストを手動実行するための専用 `Makefile` ターゲットの追加

### 対象外（Out of Scope）

- GitHub Actions など CI への組み込み（本テストは手動実行専用）。
- `make test-integration`（既存の統合テスト一括実行）への組み込み。
- 週次サマリー・警告・システムエラーなど、failure アラート以外の通知種別の送信テスト。
- Slack 上のメッセージ内容の自動アサーション（目視確認を想定）。
- 環境変数名の TOML 設定ファイルへの追加。
- failure を持たないポリシーがアラート送信されないことの検証（既存ユニットテストの責務であり、本タスクの testdata は failure を持つ単一ポリシーのみを含む）。

### 影響を受けるコンポーネント

- **新規作成**: `cmd/tlsrpt-digest` パッケージ内の統合テストファイル。
- **直接変更**: `Makefile`。

## 3. 機能要件

### `F-001`: failure を含む TLS-RPT メールのパース

`testdata/tlsrpt_failure.eml` を読み込み、添付された TLS-RPT レポートをパースして failure の有無を判定する。

**受け入れ条件（Acceptance Criteria）**:

- `AC-01`: `testdata/tlsrpt_failure.eml` から TLS-RPT レポートをパースできること。
- `AC-02`: パース結果が failure セッションを含むと判定されること。

### `F-002`: Slack アラート送信

failure を持つポリシーについて、環境変数で指定された実 Slack Webhook にアラートを送信する。

**受け入れ条件（Acceptance Criteria）**:

- `AC-03`: 環境変数で Slack Webhook URL が指定されている場合、failure を持つポリシーごとにアラートが送信されること。
- `AC-04`: 送信されるアラートには、送信組織名・ポリシー種別・失敗セッション数・レポート期間が含まれること。
- `AC-05`: すべてのアラート送信が成功した場合はテストが成功し、いずれかの送信が失敗した場合はテストが失敗すること。
- `AC-06`: 送信に用いる通知経路・メッセージ書式・リトライ挙動は本番（fetch 実行時のアラート送信）と同一であること。
- `AC-07`: 対象とする Slack Webhook 環境変数は、本番で failure アラートが送出される既存の Webhook（error チャネル用）と同一の環境変数名とすること。
- `AC-08`: 当該環境変数が未設定の場合、テストはスキップされること。

### `F-003`: 手動実行用 make ターゲット

開発者が手動で簡単に実行できる make ターゲットを提供する。

**受け入れ条件（Acceptance Criteria）**:

- `AC-09`: 専用 make ターゲットにより、本統合テストのみを実行できること（通常の `make test` では実行されないこと）。
- `AC-10`: Slack Webhook URL を環境変数で渡して実行できること。

### `F-004`: 永続アーティファクトの非作成と実行の隔離

本テストは手動での繰り返し実行や並列実行を想定する。本番 `fetch` コマンドは解析結果を `tlsrpt.json`（store 配下の永続ファイル）に保存するが、本テストではこれを作成しないことを基本とする。仮にストレージ領域が必要な場合でも、テストごとに作成・破棄される一時ディレクトリを用い、繰り返し・並列実行で他の実行と干渉しないようにする。

**受け入れ条件（Acceptance Criteria）**:

- `AC-11`: 本テストは `tlsrpt.json` をはじめとする store の永続ファイルを、リポジトリ作業ツリーや固定パス配下に作成・残置しないこと。
- `AC-12`: 本テストがストレージ領域を必要とする場合は、テストごとに作成・破棄される一時ディレクトリ（`t.TempDir` 等）を用い、並列実行時に他テストと干渉しないこと。

---

## 4. 非機能要件

### セキュリティ

- Webhook URL はコード内にハードコードせず、環境変数からのみ取得する。
- Webhook URL は、本番と同じ許可ホスト検証を経て送信されること。

### 保守性

- 通知経路は本番コードパスを再利用し、テスト専用の送信処理を重複実装しないこと（DRY）。

---

## 5. 制約

- 使用言語は Go とする（Go 1.26 以上）。
- 本テストは専用ビルドタグで隔離し、CI および `make test` / `make test-integration` では実行しない。
- テストは、本番の通知経路を再利用できる位置（`cmd/tlsrpt-digest` パッケージ）に配置する。

---

## 6. テスト方針

### このタスク自体のテスト

本タスクが追加するテストはインテグレーションテストそのものであるため、追加のユニットテストは不要。

### 手動実行手順

専用 make ターゲットに、Slack Webhook URL を環境変数で与えて実行する。実行後、指定した Slack チャンネルに以下の内容を含むアラートが届いていることを目視確認する。

- 送信組織名（`Google Inc.`）
- ポリシー種別（`sts`）
- 失敗セッション数（`2`）
- レポート期間（開始日 2026-02-08、終了日 2026-02-09）

> 環境変数名・ビルドタグ名・make ターゲット名・通知経路の具体的な組み立て方は、アーキテクチャ設計書（`02_architecture.md`）および実装計画書（`03_implementation_plan.md`）で定義する。

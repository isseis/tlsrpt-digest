## 1. 背景と目的

### 1.1 背景

failure のないレポートは即時通知ではなく、週次サマリとしてまとめて通知する。
そのためにレポートデータを永続化する仕組みが必要である。
このタスクでは `internal/store` パッケージを実装し、JSON ファイルを使ったレポートデータの蓄積と週次サマリ用の集計を担う。

JSON ファイル方式を採用する理由：

- 標準ライブラリ（`encoding/json`）のみで実装でき、外部依存を追加しない
- TLSRPT レポートの量（週数十〜百件程度）では性能上の問題が生じない
- 単一プロセス・単一ゴルーチンのポーリングループのため、並行アクセス制御が不要

### 1.2 目的

1. **主目的**: 処理済み TLSRPT レポートデータを JSON ファイルに保存する
2. **副次的目的**: 週次サマリ生成のためのデータ集計を提供する

## 2. スコープ

### 対象範囲（In Scope）

- JSON ファイルの初期化（存在しない場合の新規作成）
- レポートデータの保存（`SaveReport`）
- 指定期間のレポート取得（`GetReportsSince`）
- データファイルパスの設定

### 対象外（Out of Scope）

- 週次サマリの生成・フォーマット（タスク 0050 で担当）
- メール本文や生の JSON の保存（サマリに必要なフィールドのみ保存）

### 影響を受けるコンポーネント

- **直接変更**: `internal/store/`（新規作成）
- **間接的影響**: `cmd/tlsrpt-digest/`（store の利用側）

## 3. 機能要件

### F-001: ストレージの初期化

JSON データファイルを初期化する。

**受け入れ条件（Acceptance Criteria）**:

1. 指定パスにデータファイルが存在しない場合、空のレコードセットで新規作成される
2. 既存のデータファイルに対して初期化を呼び出しても既存データが失われない

### F-002: レポートデータの保存

パース済みの TLSRPT レポートデータを JSON ファイルに保存する。

**受け入れ条件（Acceptance Criteria）**:

1. `TLSRPTReport` の主要フィールド（組織名、レポート期間、ポリシー種別、success/failure カウント）が保存される
2. 同一レポート（`report-id` が同一）を重複保存しても整合性が保たれる（UPSERT）
3. 書き込みはアトミックに行う（一時ファイルへ書き込み後に rename）
4. 保存に失敗した場合はエラーを返す

### F-003: レポートデータの取得

指定期間（例：過去 7 日間）のレポートデータを取得する。

**受け入れ条件（Acceptance Criteria）**:

1. 指定した開始日時以降に保存されたレポートをすべて返す
2. 対象レポートが存在しない場合、空のスライスを返す（エラーにしない）
3. 取得結果は保存したデータと一致する

---

## 4. 非機能要件

### パフォーマンス

- 週次サマリ用の取得（過去 7 日分）は 1 秒以内に完了すること
- 全件メモリ読み込み後にフィルタする実装で、想定件数（週数百件以内）では許容範囲とする

### 信頼性

- 書き込みは一時ファイルへの書き込みと rename によるアトミック操作で行い、プロセスクラッシュ時のファイル破損を防ぐ

### 保守性

- `Store` インターフェースを定義し、テスト時にインメモリ実装やモックに差し替えられること

---

## 5. 制約

- 使用言語は Go とする（Go 1.23 以上）
- 外部ライブラリは使用しない（`encoding/json`・`os` 等の標準ライブラリのみ）
- テストには `stretchr/testify` を使用し、テスト用に `t.TempDir()` でテンポラリファイルを使用する

---

## 6. テスト方針

### 単体テスト

- ストレージ初期化のテスト（ファイル新規作成・既存ファイルへの冪等性）
- 保存・取得のラウンドトリップテスト
- 重複保存（UPSERT）のテスト
- アトミック書き込みのテスト（書き込み中断時にデータが破損しないこと）
- 空結果のテスト

### 統合テスト

- テンポラリファイルを使ったエンドツーエンドのテスト

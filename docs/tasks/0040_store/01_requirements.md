## 1. 背景と目的

### 1.1 背景

failure のないレポートは即時通知ではなく、週次サマリとしてまとめて通知する。
そのためにレポートデータを永続化する仕組みが必要である。
このタスクでは `internal/store` パッケージを実装し、以下の2種類のデータ永続化を担う。

1. **メール本文の保存**（`.eml` ファイル）: トラブル時の問題解析・再処理、および単体テスト用 canned データとして活用できる
2. **集計データの保存**（JSON ファイル）: 週次サマリ生成のためのレポートフィールドを蓄積する

### 1.2 目的

1. **主目的**: 処理済み TLSRPT レポートデータを JSON ファイルに保存する
2. **副次的目的**: 受信メール原本を `.eml` ファイルとして保存し、再処理・テストを可能にする

---

## 2. スコープ

### 対象範囲（In Scope）

- JSON ファイルの初期化（存在しない場合の新規作成）
- レポートデータの保存（`SaveReport`）
- 指定期間のレポート取得（`GetReportsSince`）
- `.eml` ファイルへのメール本文保存（`SaveEmail`）
- `.eml` ファイルからのメール本文読み込み（`LoadEmails`、reprocess 用）
- データファイルパスの設定

### 対象外（Out of Scope）

- 週次サマリの生成・フォーマット（タスク 0050 で担当）
- `.eml` ファイルの保持期間管理・自動削除（将来の拡張）

### 影響を受けるコンポーネント

- **直接変更**: `internal/store/`（新規作成）
- **間接的影響**: `cmd/tlsrpt-digest/`（store の利用側）

## 3. 機能要件

### F-001: ストレージの初期化

JSON データファイルおよびメール保存ディレクトリを初期化する。

**受け入れ条件（Acceptance Criteria）**:

- **`AC-01`**: 指定パスにデータファイルが存在しない場合、空のレコードセットで新規作成される
- **`AC-02`**: 既存のデータファイルに対して初期化を呼び出しても既存データが失われない
- **`AC-03`**: メール保存ディレクトリが存在しない場合、自動的に作成される

### F-002: レポートデータの保存

パース済みの TLSRPT レポートデータを JSON ファイルに保存する。

**受け入れ条件（Acceptance Criteria）**:

- **`AC-04`**: `tlsrpt.Report` の主要フィールド（`report-id`、組織名、レポート期間、ポリシー種別、success/failure カウント）が保存される
- **`AC-05`**: 同一レポート（`report-id` が同一）を重複保存しても整合性が保たれる（UPSERT）
- **`AC-06`**: 書き込みはアトミックに行う（一時ファイルへ書き込み後に rename）
- **`AC-07`**: 保存に失敗した場合はエラーを返す

### F-003: レポートデータの取得

指定期間（例：過去 7 日間）のレポートデータを取得する。

**受け入れ条件（Acceptance Criteria）**:

- **`AC-08`**: 指定した開始日時以降に保存されたレポートをすべて返す
- **`AC-09`**: 対象レポートが存在しない場合、空のスライスを返す（エラーにしない）
- **`AC-10`**: 取得結果は保存したデータと一致する

### F-004: メール本文の保存（`.eml`）

IMAP から取得したメール原本を `.eml` 形式でファイルに保存する。

保存タイミングはパース処理より前とする。これにより、パース中にプロセスが停止した場合でもメールがローカルに残り、再処理が可能になる。

ファイル名は `{YYYYMM}/{uid}-{normalized-message-id}.eml` の形式とし、月単位でディレクトリを分けることで古いファイルの手動削除を容易にする。Message-ID はそのままではファイル名として使用できない文字（`<>`、`/`、`\` 等）を含む場合があるため、正規化を行う（`<>` の除去、`/`・`\`・`:`・`*`・`?`・`"`・`|` 等の無効文字を `_` に置換、全体を 64 文字以内に切り詰める）。

**受け入れ条件（Acceptance Criteria）**:

- **`AC-11`**: メール1件につき1ファイルが作成され、RFC 2822 形式（`.eml`）で保存される
- **`AC-12`**: ファイル名に IMAP UID と正規化した Message-ID を含む（`<>` 除去、`/` 等の無効文字を `_` に置換、64 文字以内）
- **`AC-13`**: 月単位のサブディレクトリ（例：`emails/202605/`）に保存される
- **`AC-14`**: 同一 UID のメールを再度保存しようとした場合、既存ファイルを上書きしない（冪等）
- **`AC-15`**: 保存に失敗した場合はエラーを返す

### F-005: メール本文の読み込み（reprocess 用）

指定ディレクトリ以下の `.eml` ファイルを列挙・読み込む。

**受け入れ条件（Acceptance Criteria）**:

- **`AC-16`**: 指定ディレクトリ以下のすべての `.eml` ファイルを再帰的に列挙できる
- **`AC-17`**: 各ファイルを読み込み、メール構造体として返す
- **`AC-18`**: 読み込みに失敗したファイルはエラーを記録してスキップし、処理を継続する

### F-006: UIDVALIDITY の永続化と変化検出

IMAP UID はメールボックス単位で割り当てられ、`UIDVALIDITY` 値が変化したときサーバーが UID を再割り当てしていることを示す（メールボックスの再作成・移行等で発生）。UID を含むローカルファイル名（`.eml`）が誤ったメールに対応しないよう、`UIDVALIDITY` を追跡・検証する。

`internal/imap` の `FetchMeta` は `FetchMetaResult.UIDValidity` として `UIDVALIDITY` を返す。`internal/store` はこの値を永続化し、次回実行時に比較する責務を持つ。

**受け入れ条件（Acceptance Criteria）**:

- **`AC-19`**: `SaveUIDValidity(mailbox string, v uint32)` でメールボックス単位の UIDVALIDITY を保存できる
- **`AC-20`**: `LoadUIDValidity(mailbox string) (uint32, bool, error)` で保存済みの値を取得できる（未保存の場合は `false`）
- **`AC-21`**: 呼び出し元（エントリポイント）が前回値と今回値を比較し、変化を検出した場合は WARN ログを出力する
- **`AC-22`**: UIDVALIDITY が変化した場合、既存の `.eml` ファイルと UID の対応が無効になる可能性があるため、エントリポイントは当該 fetch サイクルの `.eml` ファイルとの整合性チェックをスキップし、全件を強制ダウンロード対象とする。DB 保存は `report-id` を一意識別子とした UPSERT（`AC-05`）で行うため、同一レポートの重複保存が生じても整合性は保たれる

## 4. 非機能要件

### パフォーマンス

- 週次サマリ用の取得（過去 7 日分）は 1 秒以内に完了すること
- 全件メモリ読み込み後にフィルタする実装で、想定件数（週数百件以内）では許容範囲とする

### 信頼性

- JSON 書き込みは一時ファイルへの書き込みと rename によるアトミック操作で行い、プロセスクラッシュ時のファイル破損を防ぐ
- `.eml` 保存はパース処理より前に行い、処理途中でプロセスが停止しても再処理できる状態を保つ

### 保守性

- `Store` インターフェースを定義し、テスト時にインメモリ実装やモックに差し替えられること
- 保存した `.eml` ファイルを `testdata/` にコピーすることで、単体テストの canned データとして利用できること

---

## 5. 制約

- 使用言語は Go とする（Go 1.26 以上）
- プロダクションコードでは外部ライブラリを使用しない（`encoding/json`・`os` 等の標準ライブラリのみ）
- テストコードでは `stretchr/testify` を使用してよい。テスト用に `t.TempDir()` でテンポラリディレクトリを使用する

---

## 6. テスト方針

### 単体テスト

- ストレージ初期化のテスト（ファイル新規作成・既存ファイルへの冪等性）
- 保存・取得のラウンドトリップテスト
- 重複保存（UPSERT）のテスト
- アトミック書き込みのテスト（書き込み中断時にデータが破損しないこと）
- 空結果のテスト
- `.eml` 保存のテスト（ファイル名規則・ディレクトリ構造・冪等性）
- `.eml` 読み込みのテスト（`testdata/` の実際のメールファイルを canned データとして使用）

### 統合テスト

- テンポラリディレクトリを使ったエンドツーエンドのテスト

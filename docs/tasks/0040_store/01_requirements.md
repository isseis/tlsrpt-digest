## 1. 背景と目的

### 1.1 背景

failure のないレポートは即時通知ではなく、週次サマリとしてまとめて通知する。
そのためにレポートデータを永続化する仕組みが必要である。
このタスクでは `internal/store` パッケージを実装し、SQLite を使ったレポートデータの蓄積と週次サマリ用の集計を担う。

### 1.2 目的

1. **主目的**: 処理済み TLSRPT レポートデータを SQLite に保存する
2. **副次的目的**: 週次サマリ生成のためのデータ集計クエリを提供する

---

## 2. スコープ

### 対象範囲（In Scope）

- SQLite データベースの初期化・スキーマ作成
- レポートデータの保存（`SaveReport`）
- 指定期間のレポート取得（`GetReportsSince`）
- データベースファイルパスの設定

### 対象外（Out of Scope）

- 週次サマリの生成・フォーマット（タスク 0050 で担当）
- メール本文や生の JSON の保存（サマリに必要なフィールドのみ保存）

### 影響を受けるコンポーネント

- **直接変更**: `internal/store/`（新規作成）
- **間接的影響**: `cmd/tlsrpt-digest/`（store の利用側）

## 3. 機能要件

### F-001: データベース初期化

SQLite データベースファイルを作成・初期化し、必要なテーブルを作成する。

**受け入れ条件（Acceptance Criteria）**:

1. 指定パスにデータベースファイルが存在しない場合、新規作成される
2. 既存のデータベースファイルに対して初期化を呼び出しても既存データが失われない
3. 必要なテーブル（`reports`）が存在しない場合、自動的に作成される

### F-002: レポートデータの保存

パース済みの TLSRPT レポートデータを SQLite に保存する。

**受け入れ条件（Acceptance Criteria）**:

1. `TLSRPTReport` の主要フィールド（組織名、レポート期間、ポリシー種別、success/failure カウント）が保存される
2. 同一レポート（`report-id` が同一）を重複保存しても整合性が保たれる（UPSERT）
3. 保存に失敗した場合はエラーを返す

### F-003: レポートデータの取得

指定期間（例：過去 7 日間）のレポートデータを取得する。

**受け入れ条件（Acceptance Criteria）**:

1. 指定した開始日時以降に保存されたレポートをすべて返す
2. 対象レポートが存在しない場合、空のスライスを返す（エラーにしない）
3. 取得結果は保存したデータと一致する

---

## 4. 非機能要件

### パフォーマンス

- 週次サマリ用のクエリ（過去 7 日分）は 1 秒以内に完了すること

### セキュリティ

- SQL インジェクション対策のため、クエリはプレースホルダーを使用する

### 保守性

- `Store` インターフェースを定義し、テスト時にインメモリ実装やモックに差し替えられること

---

## 5. 制約

- 使用言語は Go とする（Go 1.23 以上）
- SQLite ドライバには `modernc.org/sqlite`（CGO 不要）を優先して検討する
- テストには `stretchr/testify` を使用し、テスト用に `t.TempDir()` でテンポラリ DB を使用する

---

## 6. テスト方針

### 単体テスト

- データベース初期化のテスト
- 保存・取得のラウンドトリップテスト
- 重複保存（UPSERT）のテスト
- 空結果のテスト

### 統合テスト

- テンポラリ SQLite ファイルを使ったエンドツーエンドのテスト

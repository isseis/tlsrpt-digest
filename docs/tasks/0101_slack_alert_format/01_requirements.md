## 1. 背景と目的

### 1.1 背景

task 0100（Slack 通知インテグレーションテスト）の実行確認として実 Slack Webhook にアラートを送信したところ、現在のフォーマットに以下の問題が判明した。

1. **視認性の問題**: ヘッダと値の表示で列位置が合わず、複数ポリシーがある場合に同一ヘッダが繰り返される。
2. **情報不足（失敗詳細）**: 失敗セッション総数のみ表示しており、どのような TLS エラーが発生しているか Slack 上で判断できない。
3. **情報不足（元データ特定）**: 元の TLSRPT レポートを特定する手がかり（`report-id` 等）がなく、受け取った開発者が元データを探せない。

### 1.2 目的

1. **主目的**: 即時アラートをアクション可能な通知にする。受け取った開発者が、原因究明・対応の判断に必要な情報をアラート内で把握できるようにする。
2. **副次的目的**: 表示レイアウトを整理し、視認性を改善する。

---

## 2. スコープ

### 対象範囲（In Scope）

- 即時アラート通知（`failure_session_count > 0` の場合に送信されるアラート）の表示内容とレイアウトの改善。
- 失敗詳細（RFC 8460 `failure-details`）情報の追加表示。
- 元データ特定情報（`report-id`、レポート期間）の追加表示。

### 対象外（Out of Scope）

- 週次サマリー通知のフォーマット変更。
- システムエラー通知・フェッチ警告通知のフォーマット変更。
- 1 通の通知に収まらない場合に複数メッセージへ分割送信する仕組み（将来検討）。
- ストア上の JSON ファイルへの直接リンク（将来検討）。

### 影響を受けるコンポーネント

- **直接変更**: `internal/notify`（アラートの通知ペイロード生成および型定義）
- **間接的影響**: `cmd/tlsrpt-digest`（アラートに渡すデータの供給経路）

> 注: `failure-details` と `report-id` は現状アラートのデータ構造に含まれておらず、供給元から通知生成までのデータ経路の拡張を伴う。具体的な経路と設計は `02_architecture.md` で扱う。

## 3. 機能要件

### `F-001`: アラート表示内容の整理

アラートを、影響を受けた組織の概要と、失敗したポリシーごとの情報に整理して表示する。複数のポリシーが存在しても、同一ヘッダの繰り返しなく、ポリシーごとに情報をまとまった単位で提示する。

**受け入れ条件（Acceptance Criteria）**:

- `AC-01`: アラートの先頭に、影響を受けた組織数を含む概要を表示する（例：`⚠️ TLS Failures – 2 organizations affected`）。
- `AC-02`: 失敗した各ポリシーについて、組織名・ポリシータイプ・失敗セッション総数・レポート期間を表示する。
- `AC-03`: 失敗したポリシーが複数ある場合、すべてのポリシーが通知に含まれる。ただし Slack のメッセージサイズ制限（AC-14）により詳細表示できないポリシーがある場合は、当該ポリシーの件数・合計失敗セッション数を overflow summary として通知内に表示し、情報の欠落なく通知が完結することを保証する。
- `AC-04`: ポリシーごとの情報は、ヘッダ（項目名）が複数ポリシー間で重複表示されない形で提示される。

### `F-002`: 失敗詳細の表示

各ポリシーについて RFC 8460 の `failure-details` 情報を表示し、どのような TLS エラーが発生しているかを通知内で把握できるようにする。表示対象は、受け取った側が対応を判断するのに有用なフィールドに限定する。

| フィールド | 表示条件 | 目的 |
|---|---|---|
| `result-type` | 必須 | 失敗の種類（何を直すべきか） |
| `failed-session-count` | 必須 | 当該失敗の影響規模 |
| `receiving-mx-hostname` | 値が存在する場合のみ | どの受信サーバで発生したか |
| `failure-reason-code` | 値が存在する場合のみ | より具体的なエラー要因 |

**受け入れ条件（Acceptance Criteria）**:

- `AC-05`: `failure-details` が存在する場合、各エントリについて `result-type` と `failed-session-count` を表示する。
- `AC-06`: `receiving-mx-hostname` が値を持つ場合、そのホスト名を表示する。
- `AC-07`: `failure-reason-code` が値を持つ場合、そのコードを表示する。
- `AC-08`: あるポリシーの `failure-details` エントリが 3 件以下の場合、すべてを詳細表示する。
- `AC-09`: あるポリシーの `failure-details` エントリが 4 件以上の場合、上位 3 件を詳細表示し、残りの件数を要約表示する（例：「他 N 件」）。
- `AC-10`: `failure-details` が存在しないポリシーでも、エラーや空欄の不自然な表示を生じることなく通知が成立する。

### `F-003`: 元データ特定情報の表示

アラートを受け取った開発者が元の TLSRPT レポートを特定できるよう、識別情報を表示する。

**受け入れ条件（Acceptance Criteria）**:

- `AC-11`: アラートに、対象レポートの `report-id` を表示する。
- `AC-12`: アラートに、レポート期間（開始・終了）を表示する。

## 4. 非機能要件

### セキュリティ

- `AC-13`: アラート通知の内容に機微情報を含めない。具体的には IP アドレス（`sending-mta-ip`、`receiving-ip`）、生のエラー文字列、ファイルパス、認証情報を含めない。既存のアラート設計契約（公開可能な情報のみを含む）を踏襲する。詳細は [通知セキュリティガイドライン](../../dev/developer_guide/notification_security.md) に従う。

### 堅牢性

- `AC-14`: 表示内容が Slack の 1 メッセージあたりの制限を超える場合でも、内容を切り詰めることで送信が失敗しない。どの情報を優先的に残すかは設計で定める。

### 後方互換性

- Slack Webhook エンドポイントおよび既存設定ファイル（TOML）に変更を加えない。表示ペイロードの内容・構造の変更のみで完結する。

### 保守性

- アラートのフォーマット生成は、入力に対して決定的なテスト可能な処理として実装する。

---

## 5. 制約

- 使用言語は Go とする（Go 1.26 以上）。
- 日時はすべて UTC で表示する。
- Slack Incoming Webhook が受け付けるペイロード仕様に準拠する。

---

## 6. テスト方針

各受け入れ条件に対し、最低 1 つの自動テストで振る舞いを検証する。特に以下のケースを網羅する。

- `failure-details` の件数が 0 / 1〜3 / 4 件以上のポリシー（AC-08、AC-09、AC-10）。
- `receiving-mx-hostname`・`failure-reason-code` の値の有無の組み合わせ（AC-06、AC-07）。
- 複数組織・複数ポリシーを含むアラート（AC-01、AC-03、AC-04）。
- 機微情報が通知内容に出力されないこと（AC-13）。
- 内容が制限を超える場合の切り詰め（AC-14）。

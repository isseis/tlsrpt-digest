# ADR-0002: 機密情報（IMAP 認証情報・Slack Webhook URL）の環境変数管理

| 項目 | 内容 |

| ステータス | 採択 |

| 関連タスク | 0060_config, 0070_entrypoint |

---

## 1. コンテキスト

tlsrpt-digest は以下の機密情報を必要とする。

| 情報 | 用途 |
|---|---|
| IMAP ユーザ名 | メールサーバーへの認証 |
| IMAP パスワード | メールサーバーへの認証 |
| Slack Webhook URL（成功系） | 定期サマリの通知先 |
| Slack Webhook URL（エラー系） | アラート・エラーの通知先 |

これらをどこで管理するか（設定ファイル vs 環境変数 vs その他）を決定する必要がある。

本システムは systemd timer または cron による one-shot 実行を前提としており、デーモン化や対話的な入力は行わない。

---

## 2. 設計上の考慮事項

### 設定ファイル（TOML）に含めた場合のリスク

TOML ファイルはアプリケーションの動作パラメータ（接続先ホスト、保持期間、通知ホスト名など）を管理する。このファイルをバージョン管理（Git 等）に含めることで、チームメンバーが設定を共有・レビューしやすくなる。

一方で、パスワードや Webhook URL を同じファイルに含めると以下のリスクがある。

- 誤ってリポジトリにコミットされた場合、全履歴から削除するコストが高い
- ファイルの権限設定ミス（`chmod 644` 等）で他ユーザに読まれる
- ログ出力・エラーメッセージへの機密情報の漏洩リスクが高まる（パースエラー時に内容が表示される等）

### `notify.slack.allowed_host` を TOML に設ける理由

Webhook URL を環境変数のみで管理する場合、設定ミスや意図しない値の注入（環境変数の書き換え等）によって、通知データが攻撃者の管理するサーバーへ送信されるリスクがある。TLSRPT レポートには送信ドメイン・ポリシー種別・障害件数などの情報が含まれ、漏洩すると組織のメール送信インフラに関する情報が外部に流出する。

`notify.slack.allowed_host` を TOML に明示することで、アプリ起動時に両 Webhook URL のホスト部と照合し、想定外の送信先へのリクエストをアプリケーション層でブロックできる。TOML ファイルはバージョン管理・レビューの対象となるため、許可ホストの変更が明示的な設定変更として記録される。

### 環境変数を使う場合の特性

- systemd の `EnvironmentFile=` ディレクティブにより、`/etc/tlsrpt-digest/secrets.env` のような専用ファイルで管理できる
- secrets.env は TOML 設定ファイルとは別ファイルとして異なるパーミッション（`0600`）で管理できる
- コンテナ環境（Docker、Kubernetes）では環境変数によるシークレット注入が標準的な方法である
- `slog` ベースのデバッグログに誤って記録されにくい（`config.Secret` 型によるリダクション。既存の `internal/config/secret.go` 参照）
- cron でも `EnvironmentFile` 相当のアプローチ（実行前に `. secrets.env` を source する等）で対応できる

## 3. 検討した選択肢

### 選択肢 A: 設定ファイル（TOML）に含める

IMAP ユーザ名・パスワード・Webhook URL を TOML ファイルに記載する。

**却下理由**:

- バージョン管理への誤コミットリスクが高い
- 設定ファイルとシークレットのアクセス権限を分離できない
- パースエラー時に機密値がエラーメッセージに含まれる可能性がある（既存の `internal/config.Load` は `fmt.Errorf("config: decode failed: %w", err)` でエラーをラップしており、TOML のデコードエラーには値が含まれ得る）

### 選択肢 B: 環境変数で管理する（採択）

IMAP 認証情報と Webhook URL を環境変数で渡す。設定ファイルには接続先の検証に必要な非機密情報（`notify.slack.allowed_host`）のみを記載する。

**採択理由**: セクション 2 に記載のとおり、機密情報のライフサイクルを設定ファイルから分離することでリスクを低減できる。systemd の `EnvironmentFile=` により、既存の運用ツール（systemd-creds 等）との統合も可能である。

### 選択肢 C: 外部シークレット管理システム（HashiCorp Vault 等）を使う

実行時に Vault 等のシークレット管理システムから取得する。

**却下理由**: 本システムのユースケース（個人・小規模チームの TLSRPT 監視）に対してオーバースペックである。外部サービスへの依存が増え、可用性要件が高まる。YAGNI 原則に照らして現時点では採用しない。

### 選択肢 D: 起動時の対話入力

プログラム起動時にパスワードを標準入力から受け取る。

**却下理由**: systemd timer・cron による自動実行（非対話環境）を前提としているため採用できない。

---

## 4. 決定

機密情報（IMAP ユーザ名・パスワード・Slack Webhook URL）を環境変数で管理する。

設定ファイル（TOML）には以下の情報のみを記載する。

| 項目 | 設定先 | 理由 |
|---|---|---|
| IMAP ホスト名・ポート・メールボックス名 | TOML | 機密ではなく、環境間で共有・バージョン管理しやすい |
| `notify.slack.allowed_host` | TOML | 成功系・エラー系 Webhook URL の送信先ホスト検証用。両 URL はホスト部が一致することを前提とし、このホスト名が共通の許可済みホストとなる（機密を含まない） |
| IMAP ユーザ名・パスワード | 環境変数 | 機密情報 |
| Slack Webhook URL | 環境変数 | 機密情報 |

systemd 環境では `EnvironmentFile=/etc/tlsrpt-digest/secrets.env` で管理する。secrets.env は `0600` 権限で保護する。

## 5. 結果として生じるトレードオフ

| 得られるもの | 失うもの |
|---|---|
| 設定ファイルをリポジトリで管理しても機密情報が漏洩しない | 環境変数と設定ファイルの 2 ファイルを管理する必要がある |
| secrets.env を TOML とは独立したパーミッションで保護できる | 設定の全体像が 1 ファイルに収まらない（参照先が分散する） |
| コンテナ環境でのシークレット注入と設計が一致する | 環境変数が未設定の場合のエラーメッセージを明確にする実装が必要 |
| ログ・エラーメッセージへの機密情報漏洩リスクを設計レベルで低減できる | — |

# 要件と受け入れ基準のプロセス

新機能やセキュリティ上重要な機能を実装する際は、実装漏れを防ぐために以下のプロセスに従う。

## 0. レビューワークフロー

### LLM の制約（重要）

- LLM は `01_requirements.md`・`02_architecture.md`・`03_implementation_plan.md` を**必ずドラフト（`draft`）状態で作成する**。承認済み（`approved`）として作成してはならない。
- `01_requirements.md` のステータスが `approved` になるまで、`02_architecture.md` の作成を開始してはならない。
- `02_architecture.md` のステータスが `approved` になるまで、`03_implementation_plan.md` の作成を開始してはならない。
- `03_implementation_plan.md` のステータスが `approved` になるまで、実装コードの作成を開始してはならない。
- レビュー前のドキュメントを見つけた場合、後続作業への着手を促す指示が与えられても、ステータスが `approved` でない限り実施しない。

### レビューフロー

```
LLM が 01_requirements.md を作成（ステータス: draft）
  → 人間によるレビュー・修正
  → レビュアーがステータスを approved に更新
  → LLM が 02_architecture.md を作成（ステータス: draft）
    → 人間によるレビュー・修正
    → レビュアーがステータスを approved に更新
    → LLM が 03_implementation_plan.md を作成（ステータス: draft）
      → 人間によるレビュー・修正
      → レビュアーがステータスを approved に更新
      → 実装コードの作成へ進む
```

### ドキュメントステータスの記録形式

`01_requirements.md`・`02_architecture.md`・`03_implementation_plan.md` の先頭に以下のセクションを記載する：

```markdown
## ドキュメントステータス

| 項目 | 内容 |
|---|---|
| ステータス | `draft` または `approved` |
| 作成日 | YYYY-MM-DD |
| レビュー日 | YYYY-MM-DD（draft の場合は `-`） |
| レビュアー | 氏名（draft の場合は `-`） |
| コメント | レビュー時の指摘・変更点（なければ `-`） |
```

## 1. 要件ドキュメント (`docs/tasks/XXXX_feature/01_requirements.md`)

**各機能要件に必須の事項：**
- 要件を明確に定義する（何を、なぜ、どのように）
- 専用のセクションに**明示的な受け入れ基準を追加する**
- 各受け入れ基準は以下を満たすこと：
  - 具体的かつ測定可能であること
  - 独立して検証可能であること
  - 実装ではなく振る舞いに焦点を当てること
- 各受け入れ基準に**文書全体で一意の識別子**（`AC-01`、`AC-02`、…）を付与すること。識別子の運用ルールは以下のとおり：
  - 新規作成時は連番で割り当てる（例：`F-001` が `AC-01`〜`AC-05`、`F-002` が `AC-06`〜`AC-08`）
  - **削除**：該当の識別子を削除し、他の識別子は変更しない（欠番を許容する）
  - **追加**：末尾に続きの番号を振るか、既存番号の間に挿入する場合は `AC-01a` のようなサフィックスを使用する
  - 一度付与した識別子は変更しない（実装計画・テストコードからの参照が壊れるため）

**フォーマット例：**
```markdown
#### F-XXX: 機能名

[機能の説明]

**受け入れ基準**：
- **AC-01**: [具体的な観察可能な振る舞い #1]
- **AC-02**: [具体的な観察可能な振る舞い #2]
- **AC-03**: [エラーハンドリングの要件]
- **AC-04**: [セキュリティ要件]
- **AC-05**: [エッジケースの処理]
```

## 2. アーキテクチャ設計ドキュメント (`docs/tasks/XXXX_feature/02_architecture.md`)

**目的**：システム構成、コンポーネントの相互作用、設計上の決定に焦点を当てた高レベル設計。

**必須セクション：**
1. **設計の全体像**
   - 設計原則
   - Mermaid ダイアグラムによるコンセプトモデル

2. **システム構成**
   - Mermaid フローチャートによるアーキテクチャ全体像
   - コンポーネント配置
   - シーケンス図によるデータフロー
   - **Mermaid ダイアグラムスタイル**：[mermaid_reference.ja.md](mermaid_reference.ja.md) の規約に従う
   - **データのシリンダーノード**：フローチャートのデータソースには `[(data)]` 構文を使用する

3. **コンポーネント設計**
   - データ構造の拡張（インターフェース、型）
   - 高レベルのインターフェース定義
   - コンポーネントの責務

4. **エラーハンドリング設計**
   - エラー型の定義（インターフェースのみ）
   - エラーメッセージの設計パターン

5. **セキュリティ考慮事項**
   - セキュリティ設計パターン
   - Mermaid ダイアグラムによる脅威モデル
   - 通知機能を含む場合は [通知セキュリティガイドライン](notification_security.ja.md) に従う

6. **処理フロー詳細**
   - シーケンス図・フローチャートによる主要処理フロー

7. **テスト戦略**
   - ユニットテスト戦略
   - 統合テスト戦略
   - セキュリティテスト戦略

8. **実装の優先順位**
   - フェーズの分解
   - 順序付けられた実装ステップ

9. **将来の拡張性**
   - 将来の機能拡張に向けた設計上の考慮事項

**コンテンツガイドライン：**
- **高レベル設計に集中する**：ダイアグラムと自然言語による説明を使用する
- **コード例**：高レベルのコードのみ記載する（インターフェース、型定義、エラー型）
- **実装詳細を避ける**：具体的なコードは設計ドキュメントではなく実装に記載する
- **言語**：日本語（デフォルト）
- **フォーマット**：Mermaid ダイアグラムを含む Markdown

**参照**：`docs/tasks/0000_template/02_architecture.md`

## 3. 実装計画 (`docs/tasks/XXXX_feature/03_implementation_plan.md`)

**目的**：アクション可能なタスクとチェックボックスで実装の進捗を追跡する。

**必須セクション：**
1. **実装概要**
   - 目的
   - 実装原則

2. **実装ステップ**
   - アーキテクチャドキュメントから導いたフェーズで整理する
   - 各ステップには以下を含む：
     - **修正するファイル**：具体的なファイルパス
     - **作業内容**：何をするか（チェックボックス付き）
     - **成功条件**：完了を確認する方法
   - 追跡にはチェックボックス `[ ]` を使用：`- [ ] タスクの説明`
   - 完了済みはマーク：`- [x] 完了したタスク`
   - 部分的に完了した場合はマーク：`- [-] 部分的に完了（備考付き）`

3. **実装順序とマイルストーン**
   - 成果物を含むマイルストーン定義

4. **テスト戦略**
   - ユニットテストのカバレッジ目標
   - 統合テストのシナリオ
   - 後方互換性テスト

5. **リスク管理**
   - 軽減策を含む技術的リスク
   - バッファー計画を含むスケジュールリスク

6. **実装チェックリスト**
   - チェックボックス付きのフェーズごとのチェックリスト
   - 全体の完了追跡

7. **成功基準**
   - 機能的な完成度の指標
   - 品質指標（テストカバレッジなど）
   - セキュリティ検証の要件
   - ドキュメントの完成度

8. **次のステップ**
   - 実装後の活動

**コンテンツガイドライン：**
- **追跡に集中する**：進捗追跡にチェックボックスを多用する
- **重複を避ける**：コンテンツを繰り返すのではなく他のドキュメントを参照する
  - アーキテクチャダイアグラムや設計詳細を複製しない
  - 「設計の詳細については 02_architecture.md のセクション 3.2 を参照」のように参照する
- **アクション可能なタスク**：各チェックボックスは具体的かつ完了可能なアクションを表す
- **実装中に更新する**：タスクをリアルタイムで完了済みにマークする
- **AC トレーサビリティ**：各 AC をそれを証明するテストにマッピングする明示的な「受け入れ基準の確認」セクションを含める（§4 参照）
- **言語**：日本語（デフォルト）

**参照**：`docs/tasks/0000_template/03_implementation_plan.md`

## 4. 受け入れテスト

**適切なテストカバレッジを作成する：**
- テストは標準テストファイル（`*_test.go`）に配置する
- テスト対象に基づいた通常のテスト命名規則に従う
- ユニットテスト、統合テスト、または適切な任意の種類のテストが使用可能
- 各受け入れ基準に最低1つのテストが必要
- テストはハッピーパスだけでなく実際の振る舞いを検証すること
- 実装計画でテストを受け入れ基準にリンクする

**実装計画でのトレーサビリティ：**
`03_implementation_plan.md` で各受け入れ基準を検証するテストを文書化する：

```markdown
**AC-01: [最初の受け入れ基準]**
- テスト場所：`internal/package/subpackage_test.go::TestFunctionName`
- 実装：`internal/package/subpackage.go:123-145`
- 検証方法：[確認方法]

**AC-02: [2番目の受け入れ基準]**
- テスト場所：`internal/package/integration_test.go::TestIntegrationScenario`
- 実装：`internal/package/another.go:67-89`
- 検証方法：[確認方法]
```

**トレーサビリティコメント付きのテスト例：**
```go
// TestFoo は [振る舞い] を検証する（要件 F-001、AC-02）。
func TestFoo(t *testing.T) {
    // 特定の基準を検証するテスト実装
}
```

## 5. コミット前チェックリスト

機能が完成したと見なす前に：
- [ ] 要件ドキュメントですべての受け入れ基準が定義されている
- [ ] 高レベル設計を含むアーキテクチャ設計ドキュメントが作成されている
- [ ] 実装計画が作成され、開発中に更新されている
- [ ] 実装計画に受け入れ基準の確認セクションが存在する
- [ ] 各受け入れ基準に最低1つのテストがある
- [ ] すべての受け入れテストが通過している
- [ ] セキュリティ要件が明示的にテストされている

## 6. 背景

このプロセスは、ある機能実装において重大なセキュリティギャップが発覚した後に確立された。要件には特定のファイルを改ざん検出のためのチェックサム検証の対象とする旨が明記されていたにもかかわらず、実装ではそのチェックが省略されていた。このギャップが生じた原因は以下のとおりである：

1. 要件に明示的な受け入れ基準がなかった
2. 各基準をテストに紐付ける検証フェーズがなかった
3. セキュリティ要件を具体的に検証するテストがなかった

このプロセスは、すべての機能要件に明示的な受け入れ基準を設け、各基準とそのテストのトレーサビリティを求めることで、同様のギャップの再発を防ぐ。

# 新しいプロジェクトでの Claude コマンドの再利用

このガイドは、`.claude/commands/` 配下の Claude Code コマンドが再利用のためにどう構成されているか、そして新しいプロジェクトを立ち上げる際にどのファイルをコピーして編集すべきかを正確に説明する。要件 → アーキテクチャ → 計画 → PR → 実装という同じワークフローを使いたい、新規リポジトリを立ち上げる人を対象に書かれている。

## 3 つの層

コマンドのあらゆる内容は、3 つの層のいずれかに属する。この構成は、移植時に触れるファイルをできるだけ少なくするために存在する。

| 層 | 意味 | 変更が必要になる条件 |
|---|---|---|
| 1. プロジェクト非依存 | どのプロジェクトでも無改変で動く | なし |
| 2a. プロセス規約 | `docs/tasks/` の方法論そのもの | 新しいプロジェクトが異なる要件・設計・計画・PR ワークフローを採用する場合 |
| 2b. 技術スタック規約 | Go + Makefile + `cmd/`/`internal/` という前提 | 新しいプロジェクトが Go でない、またはビルドツールやレイアウトが異なる場合 |
| 3. ドメイン固有 | tlsrpt-digest のドメイン知識と例 | 常に |

設計上の要となるのは、**すべての層 2 と層 3 の値を 1 つのファイル `.claude/commands/_context.md` に集約している**点である。コマンド本体はそれらの値をハードコードせず名前で参照するため、移植は主に `_context.md` とそれが指すガイドドキュメントを書き換える作業になる。

## ファイルマップ

```
.claude/commands/
  _context.md                       # 層 2 + 3: すべてのプロジェクト固有の値（単一の信頼できる情報源）
  _lib/
    review-subagent-pattern.md      # 層 1: 5 つのコマンドが使う共有の批判的レビュー手順
  fixpr.md                          # 層 1: GitHub PR レビュースレッドのトリアージ（gh CLI / GraphQL）
  mktrans.md                        # 層 1: 日本語⇄英語のドキュメント翻訳ワークフロー
  mkarch.md                         # 層 2: 02_architecture.md を作成する
  mkplan.md                         # 層 2: 03_implementation_plan.md を作成する
  mkplan2.md                        # 層 2: 計画内に PR 境界を設計する
  runplan.md                        # 層 2: 計画をフェーズごとに実装する
```

`_context.md` と `_lib/*.md` は呼び出し可能なコマンドではない。これらはコマンドが実行時に読み込むデータファイルとライブラリファイルである。先頭のアンダースコアは、それらがコマンドではなく補助ファイルであることを示す。

### 間接参照の仕組み

Claude Code のコマンドには、ネイティブな include 機構がない。代わりに、各コマンドは補助ファイルを読むよう指示する一文で始まる。例えば次のとおりである。

> `.claude/commands/_context.md` を読むこと。…このコマンドがそのようなパスや値を指す箇所では、`_context.md` のエントリを正本として扱うこと。

エージェントは実行開始時にそれらのファイルを読み、値を代入する。これにより、プロジェクト固有の内容を 1 か所にまとめたまま、コマンド本体を汎用的に保てる。

## 各層に含まれる内容

**層 1（そのままコピーする）:**
- `fixpr.md` — PR レビュースレッドのワークフロー全体。`_context.md` から取得するのはビルドチェックのコマンド名（`make lint`、`make test`）のみである。
- `mktrans.md` — 翻訳ワークフロー全体。`_context.md` から取得するのは用語集のパスとデフォルトの翻訳方向のみである。
- `_lib/review-subagent-pattern.md` — `mkarch`、`mkplan`、`mkplan2`、`mktrans`、`runplan` が共有する、ペルソナ / ファイル / 評価基準 / 出力 / 3 パスのレビューループ。

**層 2（コマンドは残し、`_context.md` を書き換える）:**
- `mkarch.md`、`mkplan.md`、`mkplan2.md`、`runplan.md` — 方法論のパイプライン。その構造は汎用的であり、タスクルート、ドキュメント名、ステータス値、ガイドパス、ビルドチェック、ソースレイアウト、PR マーカーのラベルはすべて `_context.md` から取得する。

**層 3（書き換えるか削除する）:**
- 条件付きセキュリティガイドの参照と、`mkplan.md` / `runplan.md` に埋め込まれたドメイン例（ULID テスト ID、`--dry-run` の副作用、IMAP のティアダウン、`recover --mode` など）。`_context.md` はこれらがどこに現れるかを列挙しているため、検索して置き換えられる。

## 移植チェックリスト

新しいリポジトリでコマンドを再利用するには、次の手順に従う。

1. **ファイルをコピーする。** `.claude/commands/` ディレクトリ全体を新しいリポジトリへコピーする。

2. **ワークフローを決める（層 2a）。** 新しいプロジェクトが同じ要件 → アーキテクチャ → 計画 → PR のパイプラインを使うなら、方法論コマンドはそのまま残す。そうでなければ調整するが、多くのチームはパイプラインを維持し値だけを変更する。

3. **`_context.md` を書き換える。**
   - **プロセス規約**: タスクルート、ガイドパス、ドキュメント名、ステータス値、ドキュメント言語、用語集のパス、PR マーカーの規約を設定する。
   - **技術スタック規約**: ビルドチェック、ソースレイアウト、テストヘルパーの配置を設定する。新しいプロジェクトが Go でない場合は、Go 固有のエントリ（`make` ターゲット、`//go:build test`、`cmd/`/`internal/` など）を新しいスタックの相当物に置き換える。
   - **ドメイン固有**: 条件付きガイドのエントリとドメイン例を、新しいプロジェクトのドメインのものに置き換えるか削除する。

4. **コマンドが参照するガイドドキュメントを作成する。** `_context.md` は、要件プロセスガイド、タスク識別ガイド、テスト構成ガイド、Mermaid リファレンスといったガイドを指す。これらを新しいプロジェクト向けに執筆する（またはコピーして適応させる）。方法論コマンドはこれらが存在することに依存している。

5. **補助ファイルを作成する。** `mktrans.md` を残す場合は、`_context.md` に設定したパスに翻訳用語集を作成する。`docs/tasks/` のパイプラインを残す場合は、タスクテンプレートディレクトリを作成する。

6. **必要なら `/mkpr` の参照を修正する。** `runplan.md` は PR 境界スキルを名前で呼び出す。その名前が自分の構成における実際のコマンドファイルと一致することを確認する（このリポジトリではコマンドファイルは `mkplan2.md` である）。

7. **コマンドを 1 つスモークテストする。** 試験用タスクに対してコマンドを 1 つ（例えば `/mkarch`）実行し、それが `_context.md` を読み、パスを解決し、期待どおりに動作することを確認する。

## まとめ

この再構成の後、プロジェクトごとの移植コストは次の箇所に集約される。

- `_context.md` — すべてのプロジェクト固有の値を保持する 1 つのファイル。
- 開発者ガイドディレクトリ配下のガイドドキュメント — コマンドが埋め込むのではなく参照する、プロジェクトの知識ベース。

層 1 のファイル（`fixpr.md`、`mktrans.md`、`_lib/review-subagent-pattern.md`）は、無改変で引き継げる。

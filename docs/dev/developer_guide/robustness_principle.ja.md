# 堅牢性の原則

## 概要

このプロジェクトは、**堅牢性の原則**（Postel の法則とも呼ばれる）をシステム全体を通じた設計指針として採用する。

> **送るものには保守的であれ、受け取るものには寛容であれ。**
>
> — Jon Postel, RFC 793 (TCP)

## 実践的な意味

### 送るもの（出力）には保守的に

このシステムが出力を生成するとき — Slack 通知の送信、JSON ファイルの書き込み、レポートデータのフォーマット — 関連する仕様に厳密に従わなければならない。曖昧、非標準、または仕様に不足のある出力を生成してはならない。

**例:**
- TLSRPT アラートメッセージには、要件で定義された必須フィールドをすべて含めなければならない
- `internal/store` が書き込む JSON ファイルは、有効で正しい形式の JSON でなければならない
- 送信メールレポート（将来）は、正しい MIME タイプと構造を使用しなければならない

### 受け取るもの（入力）には寛容に

外部システムからデータを受け取るとき — メールサーバー、TLSRPT レポートの送信者、オペレーターが編集した設定ファイル — 厳格なパーサーであれば拒否するような合理的なばらつきを許容する。

**例:**
- TLSRPT レポートの添付ファイルを識別するとき、`Content-Type: application/tlsrpt+gzip` / `application/tlsrpt+json`（RFC 8460 MUST）を優先するが、`application/octet-stream` のような汎用タイプを使用する送信者に対しては、ファイル名拡張子（`.json.gz` / `.json`）にフォールバックする
- オペレーターが編集した TOML 設定を読み込むとき、サイレントな失敗ではなく、役立つエラーメッセージを提供する

## コードベース内での適用箇所

| コンポーネント | 保守的（出力） | 寛容（入力） |
|---|---|---|
| `internal/tlsrpt` | `*Report` を返す前に必須フィールドを検証する | `ParseGzip`/`ParseJSON` は上流の送信者によらず適切に圧縮されたデータを受け入れる |
| `internal/notify` | 厳密にフォーマットされた Slack メッセージを送信する | — |
| `cmd/tlsrpt-digest` | — | Content-Type を優先し、ファイル名フォールバックで TLSRPT 添付を振り分ける |

## 適用しないケース

この原則の「寛容」な側面は、**外部の制御できない入力**（他組織のメールサーバーからのデータ）に適用する。以下には適用**しない**：

- このシステムが生成し、このシステム自身が読み戻すデータ（内部契約は厳格に保つ）
- セキュリティ境界：セキュリティ脆弱性を防ぐ検証は絶対に緩めない（例：zip bomb 対策の展開サイズ制限は、送信者によらず厳格に維持する）
